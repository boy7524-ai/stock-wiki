# Stock Wiki — 매매일지 + 종목 비교 Dashboard 설계

작성일: 2026-05-28
대상 리포: `boy7524-ai/stock-wiki`

## 1. 목적 & 배경

stock-wiki는 Obsidian vault + GitHub 동기화 + 매일 21시(KST) 클라우드 routine으로 12종목 뉴스를 자동 리서치하는 구조다. 종목별 노트와 일일 브리핑은 잘 쌓이지만 다음 두 가지가 부족하다.

1. **종목 간 비교**가 어렵다 — 12종목의 성과·밸류에이션·리스크를 한 화면에서 비교·정렬할 수 없다.
2. **매매일지가 없다** — 본인 매수·매도 가격과 수익률을 체계적으로 기록·집계할 수단이 없다.

본 spec은 두 기능을 하나의 통합 시스템으로 추가한다. 핵심 원칙: **사용자는 거래 한 줄만 기록하고, 나머지 모든 집계·갱신은 21시 routine이 책임진다.**

## 2. 아키텍처 & 파일 구조

```
TEST/
├─ 10_Summary/
│  └─ Dashboard.md           ← routine이 매일 재생성 (사용자 편집 금지)
├─ 20_Stocks/
│  └─ <종목>.md              ← 본문은 기존 유지, frontmatter 추가 (routine이 갱신)
├─ 30_Daily_Log/
│  └─ YYYY-MM-DD_Briefing.md ← 기존 그대로
└─ 40_Trades/
   └─ 매매일지.md             ← 사용자가 거래 row 추가 (유일한 수동 입력 지점)
```

derived view (`Dashboard.md`, `매매일지.md`의 보유 포지션·실현손익 섹션, `20_Stocks/*.md`의 frontmatter)는 모두 routine이 소유한다. 사용자가 편집해야 하는 곳은 `40_Trades/매매일지.md`의 "거래 내역" 표뿐이다.

## 3. 데이터 흐름 (21시 routine)

```
1) 12종목 뉴스 리서치 (기존)
       ↓
2) 종목별 종가·PER·PER_forward·목표가 fetch (신규)
       ↓
3) 20_Stocks/*.md frontmatter 갱신 (신규)
       ↓
4) 40_Trades/매매일지.md 거래 내역 파싱 → 평단가·보유수량·실현손익·미실현 계산 (신규)
       ↓
5) 10_Summary/Dashboard.md 두 개 표 재생성 (신규)
       ↓
6) 40_Trades/매매일지.md 보유 포지션·실현손익 섹션 갱신 (신규)
       ↓
7) 30_Daily_Log/YYYY-MM-DD_Briefing.md 생성 (기존)
       ↓
8) commit & push (기존)
       ↓
9) 로컬: 다음 Claude Code 세션 시작 시 SessionStart 훅이 git pull --rebase --autostash
```

## 4. 매매일지 스키마 (`40_Trades/매매일지.md`)

```markdown
# 매매일지

> 매수/매도 발생 시 아래 "거래 내역" 표 맨 아래에 한 줄 추가. 나머지는 routine 자동.

## 거래 내역
| 날짜 | 종목 | 구분 | 수량 | 단가 | 수수료 | 메모 |
|------|------|------|------|------|--------|------|
| 2026-05-15 | [[SK하이닉스]] | 매수 | 10 | 230000 | 1000 | 1Q26 호실적 확인 |
| 2026-05-27 | [[SK하이닉스]] | 매도 | 5  | 250000 | 2500 | 부분 이익실현 |

<!-- 아래 섹션은 routine 자동 생성 — 직접 편집 금지 -->

## 보유 포지션 (routine 자동 갱신)
| 종목 | 보유수량 | 평단가 | 매수금액 | 현재가 | 평가금액 | 미실현 손익 | 미실현 수익률 |

## 실현손익 누계 (routine 자동 갱신)
| 종목 | 실현손익 | 매도 횟수 |
```

### 컬럼 정의

| 컬럼 | 형식 | 비고 |
|------|------|------|
| 날짜 | `YYYY-MM-DD` | |
| 종목 | `[[종목명]]` | Obsidian wiki-link 강제. 종목명은 `20_Stocks/`의 파일명과 일치 |
| 구분 | `매수` 또는 `매도` | 한글 고정 |
| 수량 | 정수 | 해외주식 소수 허용은 후속 |
| 단가 | **KRW** | 해외 종목도 거래일 환율로 KRW 환산해 입력 |
| 수수료 | KRW | 거래수수료 + 매도 시 거래세 합산 |
| 메모 | 자유 텍스트 | |

### 평단가 계산 — 가중평균법

- **매수**: `새 평단가 = (이전 보유금액 + 신규 매수금액) / 새 총수량`
- **매도**: `실현손익 = (매도단가 - 평단가) × 매도수량 - 수수료`. 평단가는 유지 (부분 매도 시).
- 수수료는 단순화를 위해 평단가 계산에 포함하지 않고 실현손익에서만 차감.

## 5. 종목 노트 frontmatter (`20_Stocks/*.md`)

각 종목 노트 상단에 YAML 메타데이터를 추가한다. 본문은 기존 그대로 두고 frontmatter만 routine이 갱신한다.

```yaml
---
종목코드: "000660"
섹터: 반도체
시장: KOSPI                  # KOSPI / KOSDAQ / 일본 / 미국
통화: KRW                    # KRW / JPY / USD

# 가격 (매일 갱신)
현재가: 245000               # KRW로 통일 저장
전일대비: 1.2                # %
업데이트: 2026-05-27

# 밸류에이션 (매일 갱신)
PER: 12.3                   # Trailing 12M
PER_forward: 9.8            # Forward 12M (필수)
시가총액: 178000             # 억원

# 펀더멘털 (분기 실적 발표 후 갱신)
영업이익증가율: 38.5         # 직전 분기 YoY %

# 목표가 (증권사 컨센서스 변동 시 갱신)
목표가_컨센서스: 280000
목표가_괴리율: 14.3          # (목표가 - 현재가) / 현재가 × 100

# 리스크 플래그 (수시 갱신)
리스크:
  - "부채비율 상승"
  - "환율 민감도"
---
```

### 갱신 빈도

| 빈도 | 필드 |
|------|------|
| 매일 | `현재가`, `전일대비`, `업데이트`, `목표가_괴리율` (현재가 변동 시) |
| 뉴스/실적 발표 트리거 | `PER`, `PER_forward`, `시가총액`, `영업이익증가율`, `목표가_컨센서스`, `리스크` |
| 거의 안 바뀜 (최초 1회) | `종목코드`, `섹터`, `시장`, `통화` |

### PER 표기 규칙

Trailing PER (`PER`)을 노출하는 모든 곳(브리핑, dashboard, 본문)에서 Forward PER (`PER_forward`)도 함께 표기한다. 형식: `"12.3 / 9.8"` (Trailing / Forward).

## 6. Dashboard (`10_Summary/Dashboard.md`)

routine이 매일 재생성한다. 사용자 편집 금지.

```markdown
# 종목 Dashboard
*마지막 갱신: 2026-05-28 21:02 KST*

## 📊 포트폴리오 요약
- 총 매수금액: 5,340,000 원
- 총 평가금액: 5,612,500 원
- **미실현 손익: +272,500 원 (+5.10%)**
- 실현손익 누계: +130,000 원
- 보유 종목 수: 3 / 추적 종목 수: 12

## 💼 §1 내 포지션
| 종목 | 보유수량 | 평단가 | 현재가 | 평가금액 | 미실현 손익 | 수익률 | 비중 | 실현손익 |

## 🔍 §2 종목 펀더멘털 (12종목 전체)
| 종목 | 섹터 | 현재가 | 전일대비 | PER (T/F) | 영업이익↑ | 목표가 | 괴리율 | 리스크 |

## ⚠️ 경고
- ...
```

### 표 컬럼 — 계산 공식 & 소스

| 컬럼 | 공식 / 소스 |
|------|-------------|
| §1 보유수량 | `Σ 매수수량 - Σ 매도수량` (매매일지) |
| §1 평단가 | 가중평균 매수단가 |
| §1 평가금액 | `보유수량 × 현재가` |
| §1 미실현 손익 | `평가금액 - (보유수량 × 평단가)` |
| §1 수익률 | `(현재가 - 평단가) / 평단가 × 100` |
| §1 비중 | `평가금액 / Σ 모든 평가금액 × 100` |
| §1 실현손익 | `Σ (매도단가 - 평단가) × 매도수량 - 매도수수료` |
| §2 PER (T/F) | `"PER / PER_forward"` (둘 다 표기) |
| §2 괴리율 | `(목표가 - 현재가) / 현재가 × 100` |
| §2 리스크 | 배열을 `, `로 join, 비면 `-` |

### 정렬 & 표시 규칙

- §1: 비중 desc (포지션 큰 순). 합계 row 굵게.
- §2: 보유종목 먼저 → 미보유 종목 → 섹터 알파순.
- 수익률 ≥ +10%: 굵게 / 수익률 ≤ -10%: 굵게 + 🔻.

### 경고 섹션 — Edge cases

| 케이스 | 표시 |
|--------|------|
| 보유수량 부족 매도 | ⚠️ `[종목] 보유수량 N주 < 매도수량 M주 (날짜 YYYY-MM-DD)` |
| 매매일지에 신규 종목 (`20_Stocks/`에 노트 없음) | 📌 `[종목] — 종목 노트 생성 필요` |
| 종목 가격 fetch 실패 | 📡 `[종목] 데이터 갱신 실패 (전일 값 유지)` |
| 매매일지 row 파싱 에러 | ⚠️ `매매일지 row N: 사유` |

## 7. 가격 fetch · 통화 · 비거래일

### 소스

| 시장 | 소스 |
|------|------|
| KOSPI / KOSDAQ | Naver Finance |
| 일본 | Yahoo Finance Japan |
| 미국 ADR | Yahoo Finance US (21시 KST엔 전영업일 종가만) |

### 통화 (1단계: KRW 통일)

모든 `현재가`·`단가` 필드는 KRW로 저장한다. 외화 종목은 routine이 거래일 환율로 환산. 환율 fetch 실패 시 전일 환율 사용 + 경고 플래그. native 통화 병기(`현재가_native`, `통화` 활용)는 후속 단계.

### 비거래일

- Naver/Yahoo가 종가 미제공 = 휴장 인식.
- 처리: frontmatter `업데이트` 필드 갱신 안 함 (이전 거래일 값 유지) + Dashboard 헤더에 `📅 [시장] 휴장 — 종가 미갱신` 표시.
- 한국 휴장 + 일본 정상 등 혼재 케이스도 동일 로직.

## 8. 구현 Phase

### Phase 1 — Bootstrap (1회)
1. `20_Stocks/*.md` 12개에 frontmatter 추가 (`종목코드`, `섹터`, `시장`, `통화`만 사용자 검수. 나머지는 빈 값 → routine 첫 실행이 채움)
2. `40_Trades/매매일지.md` 생성 — 거래 내역 표 헤더 + 사용법 설명
3. `10_Summary/Dashboard.md` placeholder — "첫 routine 실행 후 자동 생성됩니다"

### Phase 2 — Routine 업데이트 (핵심)
4. 21시 routine prompt에 4단계 추가 (가격 fetch → frontmatter merge → 매매일지 파싱 → Dashboard 생성)
5. 즉시 1회 수동 실행해 정상 동작 검증

### Phase 3 — 운용 시작
6. 사용자가 매매일지에 실거래 row 입력 시작 (선택)
7. 매일 21시 routine 자동 갱신, SessionStart 훅이 다음 세션 시작 시 pull

### Phase 4 — 후속 (별도 spec)
- 통화 native 병기
- 분기 실적 발표 자동 감지 → 펀더멘털 필드 갱신
- 목표가 도달·수익률 임계 초과 시 알림 섹션
- Dataview 쿼리 옵션 추가
- 종목 추가/제거 워크플로우

## 9. 위험 요소 & 완화

| 위험 | 완화 |
|------|------|
| Naver/Yahoo HTML 구조 변경 | "fetch 실패 시 전일 값 유지" 로직 + Dashboard 경고 가시화 |
| 첫 실행 부하 (12종목 × 다중 fetch) | routine 예상 실행 시간 4~5분, 21시 cron 시간 여유 충분 |
| 매매일지 마크다운 표 파싱 오류 | 파싱 실패 row 무시 + ⚠️ 섹션 기록 (전체 routine 중단 안 됨) |
| 부분 매도 후 평단가 처리 오류 | 가중평균법 명시 + 합계 row 검산 |

## 10. 비목표 (Out of Scope)

- 실시간 가격 (장중) — 일일 종가 기준으로 충분
- 매매 자동 실행 — 기록·집계 전용
- 세금 계산 (양도세·종합소득세) — 후속
- 차트 시각화 (가격 차트, 수익률 그래프) — 후속
- Obsidian Dataview 의존 — 마크다운 표가 1차 출력, Dataview는 옵션
